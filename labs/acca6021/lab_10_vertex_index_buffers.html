Lab 10 - Vertex and Index Buffers

Overview

Building on the GlCall error system from Lab 09 (GlCall Error System), this lab wraps two of the most fundamental OpenGL objects in C++ classes. The Vertex Buffer Object (VBO) holds your vertex data on the GPU - positions, colours, UVs, normals. The Index Buffer Object (IBO) holds integer indices that tell the GPU which vertices to connect into triangles, allowing a four-vertex quad to be drawn with six indices instead of six duplicated vertices. Both follow the same RAII pattern introduced in the previous lab: the constructor calls glGenBuffers and uploads data, and the destructor calls glDeleteBuffers so GPU memory is released automatically when the object goes out of scope.

Learning Objectives

By the end of this lab you will be able to:

  • Explain what a GPU buffer object is and why vertex data must be uploaded before the render loop
  • Distinguish between GL_STATIC_DRAW (data set once) and GL_DYNAMIC_DRAW (data updated per frame)
  • Implement the VertexBuffer class following the RAII pattern: generate in the constructor, delete in the destructor
  • Explain why IndexBuffer stores m_Count and how glDrawElements uses it instead of glDrawArrays
  • Use ASSERT to confirm that unsigned int and GLuint have the same size before uploading index data

Concept

When you called glBufferData directly in main.cpp, the GPU-side memory management was entirely your responsibility. You had to remember the buffer ID in a raw unsigned int, call glDeleteBuffers manually before the program exited, and repeat the same four-line setup everywhere you needed a new buffer. A Vertex Buffer Object is simply a named region of GPU memory. glGenBuffers allocates it and returns an integer handle. glBindBuffer(GL_ARRAY_BUFFER, id) makes it the current target for vertex data. glBufferData uploads bytes from a CPU pointer to that region. The GL_STATIC_DRAW hint tells the driver to place the data in fast, read-only VRAM because you will not change it. GL_DYNAMIC_DRAW tells the driver to keep the buffer in a region that supports frequent rewrites, suited for particle systems or animated geometry.

An Index Buffer Object (also called an Element Buffer Object or EBO) works like a lookup table for vertices. A quad has four unique corners but needs six vertex references to draw two triangles. Without an IBO you would repeat two vertices and waste bandwidth. With an IBO you upload four vertices to the VBO and an index array { 0, 1, 2, 2, 3, 0 } to the IBO. The GPU reads the indices and fetches vertices in that order. glDrawElements replaces glDrawArrays and takes the number of indices as its second argument - which is why the IndexBuffer class stores m_Count. The portability check ASSERT(sizeof(unsigned int) == sizeof(GLuint)) confirms that the C++ type you are using to build the index array is the same width as what OpenGL expects. On all modern desktop platforms they match, but the assertion makes the assumption explicit and will break loudly if it ever fails.

Both classes include Renderer.h in their .cpp files so every OpenGL call is wrapped in GlCall. The header files do not include Renderer.h - only the implementations need it, and keeping includes out of headers avoids cascading recompilation. The constructor takes a raw const void* for vertex data rather than a typed pointer because vertices can hold floats, unsigned bytes, or packed structs - a void pointer accepts all of them without a cast at the call site.
