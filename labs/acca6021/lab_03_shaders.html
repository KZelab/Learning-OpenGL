ACCA6021 - Graphics and Games Programming

Lab 03 - Shaders

Overview

In the First Triangle lab (Lab 02) you relied on OpenGL's built-in fixed-function pipeline to colour the triangle white. This lab replaces that with your own GLSL shader program. You will write a vertex shader and a fragment shader, store them together in a single .shader file, write a C++ parser to load them, compile both stages with error checking, and link them into a program that drives every pixel of the triangle.

The project also returns to Core Profile, which disables the fixed-function fallback entirely. From Lab 03 onward, all rendering requires an explicit shader program.

Learning Objectives

By the end of this lab you will be able to:

  • Describe the roles of the vertex shader and fragment shader within the OpenGL pipeline.
  • Write basic GLSL for both stages and explain the purpose of gl_Position and a fragment output variable.
  • Parse a custom .shader file format to separate vertex and fragment source at runtime.
  • Compile shader stages and read driver error messages using glGetShaderInfoLog.
  • Link a shader program and activate it before a draw call with glUseProgram.

Concept

Switching back to Core Profile removes the fixed-function pipeline entirely. Every draw call now requires an active shader program or nothing renders. A program contains at minimum two stages. The vertex shader runs once for each vertex the draw call submits. It receives the position from attribute slot 0 and must write a clip-space position to the built-in output gl_Position. OpenGL then determines which screen pixels the assembled triangle covers.

The fragment shader runs once for each of those covered pixels. It must write a colour to a user-declared out vec4 variable. Both shaders are written in GLSL and compiled at runtime by the GPU driver via glCreateShader, glShaderSource, and glCompileShader. Compile errors are returned as strings via glGetShaderInfoLog. The two compiled shaders are attached to a program object with glAttachShader and linked with glLinkProgram. Calling glUseProgram before a draw call activates that program for the pipeline.

Rather than managing two separate source files per shader, this project uses a single .shader file with custom section markers. The lines #shader vertex and #shader fragment are not valid GLSL. They are directives read by a C++ parser that distributes each subsequent line to the correct source string. The OpenGL compile API receives two clean GLSL strings with no trace of the markers.

Task 4 - Add the Shader Compiler with Error Checking

Copy the function below into src/main.cpp, above main() and after the parser. It compiles one shader stage and prints any GLSL error message the driver reports. This is provided in full because the error-checking API is boilerplate you will reuse unchanged in every project.

src/main.cpp
static unsigned int compileShader(unsigned int type, const std::string& source) {
    unsigned int id = glCreateShader(type);
    const char* src = source.c_str();
    glShaderSource(id, 1, &src, nullptr);
    glCompileShader(id);

    int result;
    glGetShaderiv(id, GL_COMPILE_STATUS, &result);
    if (!result) {
        int length;
        glGetShaderiv(id, GL_INFO_LOG_LENGTH, &length);
        char* message = new char[length];
        glGetShaderInfoLog(id, length, &length, message);
        std::cerr << "Shader compile error: " << message << '\n';
        delete[] message;
        return 0;
    }
    return id;
}

To test the error reporting, introduce a deliberate typo in the fragment shader (for example, remove the semicolon after FragColour = ...), run the project, and read the console output. Fix the typo before continuing.
