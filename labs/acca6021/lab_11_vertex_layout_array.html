Lab 11 - Vertex Layout and Array

Overview

The Vertex and Index Buffers lab (Lab 10) created classes that allocate raw bytes on the GPU. Raw bytes alone are meaningless to the shader - it has no way to know that bytes 0-11 are a position and bytes 12-19 are UV coordinates. This lab introduces the two classes that solve that problem. VertexBufferLayout builds a description of your vertex format by recording the type and size of each attribute. VertexArray (VAO) uses that description to call glVertexAttribPointer once during setup, and then records the result so the entire vertex format can be replayed later with a single Bind() call. Together they replace the manual glEnableVertexAttribArray loop that appeared directly in main.cpp in earlier labs.

Learning Objectives

By the end of this lab you will be able to:

  • Explain what stride and offset mean and how they describe the position of each attribute within a packed vertex
  • Use VertexBufferLayout::Push to build a vertex description and observe how m_Stride accumulates with each call
  • Explain why the VAO is called a recording device and what exactly it records during AddBuffer
  • Implement VertexArray::AddBuffer, iterating the layout elements and calling glVertexAttribPointer for each
  • Replace the manual attribute setup in main.cpp with a VertexArray and VertexBufferLayout

Concept

A vertex buffer holds a flat array of bytes. Your CPU code knows that the first 12 bytes are a 3-float position, the next 8 bytes are 2-float UV coordinates, and so on - but the GPU does not. To bridge that gap, OpenGL needs you to call glVertexAttribPointer once per attribute, telling it: the attribute index (matching layout(location = N) in the shader), how many components the attribute has, the data type, whether to normalise, the total byte distance between one vertex and the next (stride), and the byte offset from the start of the vertex to this particular attribute (offset). Without these calls, every vertex shader invocation reads garbage.

The VertexBufferLayout class captures the vertex description as a list of VertexBufferElement structs. Each element stores its GL type, component count, and normalisation flag. The Push<float>(3) call appends a three-component float element and adds 3 * 4 = 12 bytes to the running stride. Calling Push<float>(2) afterwards appends a two-component element and adds 8 more bytes. When all attributes have been pushed, GetStride() returns the total vertex size in bytes and GetElements() returns the ordered list. Template specialisations for float, unsigned int, and unsigned char cover the most common attribute types. A general template with a static assertion catches any unsupported type at compile time rather than at runtime.

A Vertex Array Object (VAO) is a recording device built into the GPU driver. While a VAO is bound, every call to glEnableVertexAttribArray and glVertexAttribPointer is saved inside that VAO. The next time you bind the VAO, the driver replays all those calls instantly - no loop, no manual stride arithmetic. VertexArray::AddBuffer exploits this by binding both the VAO and the VBO, iterating the layout elements, and calling glVertexAttribPointer once per element with the accumulated byte offset. When AddBuffer returns, the VAO remembers exactly which buffer holds the data and how to interpret each attribute. Binding the VAO in the render loop is then the only setup needed before a draw call.

src/VertexBufferLayout.h
#pragma once
#include <vector>
#include "Renderer.h"

// Describes a single vertex attribute: its GL type, component count, and whether to normalise
struct VertexBufferElement
{
    unsigned int  type;
    unsigned int  count;
    unsigned char normalised;

    // Returns the byte size of one component of the given GL type
    static unsigned int getSizeOfType(unsigned int type)
    {
        switch (type)
        {
            case GL_FLOAT:         return 4;
            case GL_UNSIGNED_INT:  return 4;
            case GL_UNSIGNED_BYTE: return 1;
            default: throw std::runtime_error("Unknown type!");
        }
        ASSERT(false);
        return 0;
    }

    VertexBufferElement(unsigned int type, unsigned int count, unsigned char normalised)
        : type(type), count(count), normalised(normalised)
    {
    }
};

// Describes the full layout of one vertex: ordered list of attributes and the total stride
class VertexBufferLayout
{
private:
    std::vector<VertexBufferElement> m_Elements;
    unsigned int m_Stride;
public:
    VertexBufferLayout() : m_Stride(0) {}
    ~VertexBufferLayout() {}

    // Generic Push - compile error for any type without a specialisation below
    template<typename t>
    void Push(unsigned int count)
    {
        static_assert(sizeof(t) == 0, "Unsupported type for VertexBufferLayout::Push.");
    }

    // Push a float attribute (e.g. Push<float>(3) for a vec3 position)
    template<>
    void Push<float>(unsigned int count)
    {
        m_Elements.push_back({ GL_FLOAT, count, GL_FALSE });
        m_Stride += count * VertexBufferElement::getSizeOfType(GL_FLOAT);
    }

    // Push an unsigned int attribute (e.g. for integer vertex IDs)
    template<>
    void Push<unsigned int>(unsigned int count)
    {
        m_Elements.push_back({ GL_UNSIGNED_INT, count, GL_FALSE });
        m_Stride += count * VertexBufferElement::getSizeOfType(GL_UNSIGNED_INT);
    }

    // Push a normalised unsigned byte attribute (e.g. packed colour 0-255 mapped to 0.0-1.0)
    template<>
    void Push<unsigned char>(unsigned int count)
    {
        m_Elements.push_back({ GL_UNSIGNED_BYTE, count, GL_TRUE });
        m_Stride += count * VertexBufferElement::getSizeOfType(GL_UNSIGNED_BYTE);
    }

    inline const std::vector<VertexBufferElement> GetElements() const { return m_Elements; }
    inline unsigned int GetStride() const { return m_Stride; }
};
