Lab 21 - AABB Player-Wall Collision

Overview

In the Maze Layout lab (Lab 20) you built a navigable first-person maze, but the camera passes through walls. This lab fixes that by introducing the Player class. Player wraps the Camera in a small Axis-Aligned Bounding Box (AABB) and resolves collisions using a technique called move and slide: the frame's WASD movement is applied one axis at a time, and any wall overlap is corrected after each axis independently. The result is that walking diagonally into a wall cancels the blocked component of the motion but keeps the free component, so the player slides smoothly along the surface rather than sticking to it.

Learning Objectives

By the end of this lab you will be able to:

  • Explain what an AABB is and how two AABB objects are tested for overlap on a single axis.
  • Implement the move-and-slide algorithm: snapshot the previous position, apply WASD input, then resolve X and Z independently.
  • Implement ResolveAxis: a 3x3 broad-phase neighbourhood check that pushes the player out of overlapping wall tiles.
  • Wire the Player class into TestMaze by delegating Update and Respawn, removing the direct camera-grounded call from the constructor.
  • Verify that the player slides along walls when moving diagonally and cannot pass through any wall tile.

Concept

Without collision, the camera moves wherever the input directs it - through walls and all. The standard game-development fix is to give the player a bounding volume and test it against the world each frame. Here the player is an AABB: a box whose sides are aligned to the world axes. An AABB is defined by a centre position and a half-extent (half the box's width) on each axis. Two AABBs overlap on one axis when the distance between their centres is less than the sum of their half-extents. If they overlap on every axis simultaneously, they are colliding.

Resolving the collision naively - just push the player back to their previous position - causes them to stick to the wall when moving diagonally. The correct approach is move and slide: save the position before input, let the camera apply this frame's movement to get the desired position, then re-apply the movement one axis at a time and resolve any overlap after each axis separately. If the X axis is blocked the player is pushed back on X, but the Z movement (which may be clear) is still applied, so the player slides along the wall face.

Testing every tile in the maze for overlap each frame would be expensive. The broad-phase optimisation exploits the fact that the player box is much smaller than one tile: only the 3x3 block of tiles around the player's current tile can possibly touch the bounding box. MazeMap::WorldToTile converts the player's world position to a tile index, and the 3x3 loop uses MazeMap::IsWall and MazeMap::TileCenter to check and position each candidate tile. Both methods are already implemented and ready to use.

  1. Add Update and ResolveAxis to src/tests/maze/Player.cpp inside the namespace test block, after Respawn.
  2. In Update:
    1. Save the camera's current position as previous.
    2. Call m_Camera->processInput(deltaTime) to apply WASD.
    3. Save the result as desired.
    4. Build resolved = previous, then apply each axis in turn: 
      resolved.x = desired.x;  ResolveAxis(resolved, 0);
resolved.z = desired.z;  ResolveAxis(resolved, 2);
resolved.y = desired.y;  // grounded camera pins this already
    5. Write resolved back with m_Camera->setPosition(resolved).
  3. In ResolveAxis:
    1. Compute the player's current tile with m_Map->WorldToTile(position, playerCol, playerRow).
    2. Loop over the 3x3 neighbourhood (row = playerRow - 1 to playerRow + 1, same for col). Skip tiles where m_Map->IsWall returns false.
    3. For each wall tile, compute the AABB overlap on X and Z: 
      float overlapX = (m_HalfExtent + wallHalf) - std::abs(position.x - wallCenter.x);
float overlapZ = (m_HalfExtent + wallHalf) - std::abs(position.z - wallCenter.z);
      If either overlap is zero or negative there is a gap - skip with continue.
    4. Push the player out along the axis parameter only. If position[axis] > wallCenter[axis], push to the positive face; otherwise push to the negative face. Add a small epsilon nudge to avoid floating-point re-entry next frame.

    void Player::ResolveAxis(glm::vec3& position, int axis) const
    {
        // Each wall tile is tileSize wide; the wall surface is tileSize/2
        // from the tile centre on each side.
        const float wallHalf = m_Map->TileSize() * 0.5f;

        // Epsilon nudge prevents floating-point error leaving the player
        // microscopically overlapping a wall surface after resolution.
        const float epsilon  = 0.001f;

        // BROAD PHASE: the player box is much smaller than a tile, so only
        // the 3x3 block of tiles around the player's current tile can
        // possibly touch the bounding box.
        int playerCol, playerRow;
        m_Map->WorldToTile(position, playerCol, playerRow);

        for (int row = playerRow - 1; row <= playerRow + 1; row++)
        {
            for (int col = playerCol - 1; col <= playerCol + 1; col++)
            {
                if (!m_Map->IsWall(col, row))
                    continue;

                glm::vec3 wallCenter = m_Map->TileCenter(col, row);

                // AABB OVERLAP TEST:
                // Two axis-aligned boxes overlap when the distance between
                // their centres is LESS than the sum of their half-extents,
                // on EVERY axis simultaneously. If either axis has a gap,
                // there is no collision and we skip this tile.
                //
                // overlap = (halfA + halfB) - |centreA - centreB|
                // overlap > 0  =>  boxes overlap on that axis
                //
                // TODO: float overlapX = (m_HalfExtent + wallHalf)
                //                        - std::abs(position.x - wallCenter.x);
                // TODO: float overlapZ = (m_HalfExtent + wallHalf)
                //                        - std::abs(position.z - wallCenter.z);
                // TODO: if (overlapX <= 0.0f || overlapZ <= 0.0f) continue;

                // PUSH OUT along the axis being resolved:
                // If the player centre is to the POSITIVE side of the wall
                // centre, push to the positive face; otherwise push negative.
                //
                // TODO: if (position[axis] > wallCenter[axis])

                //                            + wallHalf + m_HalfExtent + epsilon;
                //       else
                //           position[axis] = wallCenter[axis]
                //                            - wallHalf - m_HalfExtent - epsilon;
            }
        }
